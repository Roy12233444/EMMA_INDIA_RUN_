## 🛰️ 2. Comprehensive Class Specifications

### Class A: `ASTContextRotator`
*   **Algorithmic Concept: Geodesic Coordinate Reduction & Manifold Projection**
    *   *Mathematical Intuition:* Treat the complete file as a high-dimensional state space. When a massive file is loaded, it contains extraneous coordinates (unrelated methods) that pollute the agent's focus. We project this high-dimensional coordinate structure onto a localized, low-dimensional "active manifold" containing only the exact class/method under edit.
    *   *Mechanism:* 
        *   Rather than presenting a raw flat file, we compile the source code into a hierarchical tree of nodes.
        *   We locate the coordinates (start and end line numbers) of the active node being edited.
        *   All sibling nodes (unrelated methods) are collapsed (rotated out of view). We extract only their structural signatures (e.g. `def connect(self, ws: WebSocket) -> None:`) and append an ellipsis stub (`...`).
        *   The active node remains $100\%$ fully expanded.
        *   The combined, highly-compressed file represents the optimal geodesic path on the codebase's semantic manifold, keeping context sizes strictly under **1,500 tokens**.
        *   We encapsulate this projection inside clean XML boundary tags: `<TRANSIENT_CONTEXT id="{node_name}" type="ast_node"> ... </TRANSIENT_CONTEXT>`.

#### **Core Methods & Algorithms:**
1.  `__init__(self, file_path: str)`:
    *   Saves target filepath.
    *   Loads file, handles `utf-8` encoding.
    *   Compiles source code to an AST using `ast.parse`.
2.  `_map_nodes(self)`:
    *   Traverses the tree using `ast.walk`.
    *   Maps each `ast.FunctionDef`, `ast.AsyncFunctionDef`, and `ast.ClassDef`.
    *   Records: start line (`lineno`), end line (`end_lineno`), and sibling relationship chains.
3.  `_extract_signature(self, node: ast.AST) -> str`:
    *   Rebuilds the standard definition line with its exact argument annotations and return types (e.g. `def connect(self, websocket: WebSocket) -> None:`), appending ` ...` instead of the function body.
4.  `get_rotated_context(self, active_node_name: str) -> str`:
    *   Identifies the target function inside our lines map.
    *   Iterates through the file lines. For all sibling functions, it replaces their body lines with their signature stubs.
    *   Keeps the active function's body fully intact.
    *   Wraps the entire formatted text inside `<TRANSIENT_CONTEXT id="{node_name}" type="ast_node">` tags.

### Class B: `MutantCodeSelector`
*   **Algorithmic Concept: Evolutionary Mutation & Syntactic Fitness Selection**
    *   *Mathematical Intuition:* Treat code editing as an evolutionary survival process. Let $C = \{c_1, c_2, c_3\}$ represent a population of candidate code variations (mutants) generated by the LLM. Rather than blindly committing code, we pass the candidates through a multi-factor **Fitness Objective Function** to find the global optimum:
        
        $$\text{Fitness}(c) = \text{SyntaxCheck}(c) - \text{ParsimonyPenalty}(c) - \text{ConstraintPenalty}(c)$$
        
    *   *Mechanism:*
        *   **Syntax Check:** Compiles $c$ in-memory. If compile is successful, awards $+50.0$ points. If compilation fails (SyntaxError), assigns a massive penalty of $-100.0$ points and halts further scoring for that candidate.
        *   **Parsimony Penalty:** Deducts $0.1$ points per line of code to favor clean, elegant, and compact implementations over verbose structures.
        *   **Constraint Penalty:** Analyzes the target function signature. If a return statement is expected but completely absent from the candidate body, deducts $-30.0$ points.
        *   The algorithm automatically selects and commits the candidate that maximizes the fitness score.

#### **Core Methods & Algorithms:**
1.  `__init__(self, target_signature: str)`:
    *   Sets target signature constraints (e.g. checking if return statements are mathematically required).
2.  `parse_syntax_check(self, code: str) -> bool`:
    *   Runs `ast.parse(code)` in-memory. Returns `True` if it compiles, `False` if a `SyntaxError` is raised.
3.  `calculate_parsimony(self, code: str) -> float`:
    *   Scores code parsimony (verbosity control). Deducts $0.1$ points for every line of code to encourage compact, elegant functions.
4.  `evaluate_mutants(self, candidate_codes: list[str]) -> str`:
    *   Iterates through 3 candidate mutant strings in-memory.
    *   Grades their fitness:
        *   **AST Parse Success:** $+50.0$ points.
        *   **AST Parse Failure (SyntaxError):** $-100.0$ points.
        *   **Missing Return Check:** Deducts $-30.0$ points if the target signature expects a return but `return` is absent from the mutant body.
        *   **Parsimony Deduction:** Deducts `len(lines) * 0.1` points.
    *   Returns the candidate with the highest fitness score.

### Class C: `PageCurveEvaporator`
*   **Algorithmic Concept: Entropic Information Decay & Page Curve Compression**
    *   *Mathematical Intuition:* Modeled on the Hawking Radiation Page Curve, which demonstrates that information hoarded in a dense system must eventually evaporate (decay) to preserve balance. As active run standard output and trace logs accumulate, they peak in entropy. Once this tipping point (20 lines) is crossed, we systematically trigger "evaporation."
    *   *Mechanism:*
        *   We strip away all low-value informational lines, noise lines, and repetitive stdout prints.
        *   We scan the raw stream using high-speed regular expressions to extract only the high-value structural metrics: Exit Code Status, Exception Types, Warning counts, and Error line outputs.
        *   We condense this entire high-entropy log block into a tiny, lossless, 1-line metadata block.
        *   This recovers $98\%$ of the token footprint while keeping the agent fully informed of execution outcomes.

#### **Core Methods & Algorithms:**
1.  `__init__(self, max_lines: int = 20)`:
    *   Defines the active line threshold (Page Time limit).
2.  `evaporate_log(self, raw_stdout: str) -> str`:
    *   Checks line count. If it is under `max_lines`, returns `raw_stdout` intact.
    *   If it exceeds the limit:
        *   Runs regex queries (`re.search`) for error patterns like `ERR`, `FAIL`, `Exception`, `warning`, `exit status`, and `exit code`.
        *   Extracts the final failing trace line or error exception message.
        *   Synthesizes a 1-line metadata summary string containing line counts, error counts, warning counts, and target exit codes.
        *   Returns: `[Log Evaporated: Total Lines={lines} | Errors={errors} | Warnings={warnings} | Status={exit_code}]`.
