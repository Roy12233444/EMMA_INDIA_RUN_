# Implementation Plan: EMM-05-A2 — Mathematical Safety: Jailed Python Sandbox Subprocess

## 1. Goal Description
The purpose of this ticket is to implement a secure, resource-capped subprocess execution model for Python code mutants generated by the EMMA Cognitive Core. Currently, the sandbox in `code_generator.py` uses in-memory `exec()` with custom built-ins, which does not prevent resource exhaustion, infinite CPU loops, or process crashes.

This new architecture will:
- Spawn code mutants in a **dedicated, isolated subprocess** using Python's standard `subprocess` module.
- Enforce a strict **30-second execution timeout**.
- Enforce a strict **256MB RAM ceiling** on the subprocess.
- Map resource constraints in an OS-specific wrapper:
  - On **Windows**: Map memory limits using native Windows Job Objects via `ctypes`.
  - On **Unix/Linux**: Map memory limits using standard `resource` limits in the child process.

## 3. Detailed Design

### A. Windows Job Object Memory Cap (using `ctypes`)
Since Windows does not support `resource.setrlimit`, we must map memory limits to a Windows Job Object.
1. Create a Job Object using `CreateJobObjectW`.
2. Configure limit flags:
   - `JOB_OBJECT_LIMIT_PROCESS_MEMORY` to restrict virtual memory usage.
   - `JOB_OBJECT_LIMIT_KILL_ON_JOB_CLOSE` to ensure all child processes are killed if the handle is closed or if the parent terminates.
3. Configure `JOBOBJECT_EXTENDED_LIMIT_INFORMATION` structure with `ProcessMemoryLimit = 256 * 1024 * 1024` bytes (256MB).
4. Spawn the subprocess suspended or immediately assign it using `AssignProcessToJobObject` using the process handle.

### B. Unix Resource Limit (using `resource`)
On Unix platforms:
- Use `subprocess.Popen` with a `preexec_fn` callback.
- In the callback, call `resource.setrlimit(resource.RLIMIT_AS, (256 * 1024 * 1024, 256 * 1024 * 1024))`.

### C. Subprocess Protocol
The parent process will run the generated python script as a subprocess:
1. The subprocess receives the code to execute via `stdin` or a temp file.
2. The subprocess executes the code within a secure wrapper (with standard safe builtins).
3. The subprocess writes standard output and error back to the parent.
4. The parent catches `subprocess.TimeoutExpired` if execution goes over 30s.
5. If the subprocess exceeds 256MB RAM, the OS kernel will automatically abort/kill it. The parent detects the termination signal (non-zero/negative exit status) and reports it as an out-of-memory failure.

## 4. Verification Plan

### Automated Tests
- Create a unit/integration test suite:
  - Verify that a standard mathematical function (e.g. calculation) runs successfully.
  - Verify that a CPU loop (e.g. `while True: pass`) terminates within the timeout limit (30s) and returns a clean timeout error.
  - Verify that a memory loop (e.g. `[0] * 100_000_000`) is terminated due to exceeding the 256MB limit on both Windows and Linux.
  - Verify that illegal operations (like trying to write files or import blocked builtins) are caught at the execution stage.
